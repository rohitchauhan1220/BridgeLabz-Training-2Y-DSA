Problem 2: Time Complexity Analysis and Comparison
a)Worst-case number of comparisons (n = 10,000,000)
Algorithm A: Linear Search
In the worst case, the username is not present or is the last element.
Comparisons required = n
n=10,000,000
Worst-case comparisons = 10,000,000

Algorithm B: Binary Search
Each comparison reduces the search space by half.
Worst-case comparisons = (log₂ n)
log2(10,000,000)≈23.25⇒24 comparisons
Worst-case comparisons ≈ 24

Algorithm C: Hash Table Lookup
In the worst case (ideal hashing), lookup takes 1 comparison.
Worst-case comparisons = 1

b) Time Complexity (Big-O Notation)
Algorithm A: Linear Search
Time taken increases directly with n,  T(n)=n , Time Complexity Big-O: O(n)
Algorithm B: Binary Search
Search space halves every step , T(n)=log2​n ,Time Complexity Big-O: O(log n)  
Algorithm C: Hash Table Lookup
Lookup time is constant (on average) ,T(n)=1, Big-O: O(1)

e) When Is Sorting + Binary Search Worthwhile?
Costs
Sorting cost = O(n log n)
Binary search per query = O(log n)
Linear search per query = O(n)
Let:
k = number of searches
Linear Search Total Cost    k*n

Sort + Binary Search Total Cost    nlogn+klogn
Condition
Binary search is better when:
nlogn+klogn<kn

Final Conclusion
Linear Search: Too slow for large datasets
Binary Search: Efficient but requires sorting
Hash Table: Fastest and best choice for real-time systems
Best choice for production: Hash Table Lookup
